# 性能分析 Skill

针对 C(含 kernel)/Go/Python 网络设备后端工程的性能优化分析工具。

## 使用原则

1. **优先适配产品环境**：默认假设代码运行在目标设备或产品环境，而不是开发机。
2. **优先低扰动取证**：先使用代码静态分析和 `/proc`、`/sys` 等只读信息，再视需要升级到更重的采样手段。
3. **先给方向，再补证据**：第一轮可以先给基于代码的判断，但必须明确置信度和待验证项。
4. **默认终端输出**：优先在终端给出摘要和下一步建议；只有用户明确需要沉淀文档时，才生成完整 Markdown 报告。

## 读取路径

只在需要时读取对应参考文件，不要一次性把所有参考文档都展开。

- 需要扫描代码中的性能反模式时，读取 `references/code-patterns.md`
- 需要分析 C 用户态程序或内核模块的 CPU / 内存 / syscall / 锁问题时，读取 `references/c-profiler.md`
- 需要分析 Go 服务的 pprof / trace / goroutine / GC / mutex 问题时，读取 `references/go-profiler.md`
- 需要分析 Python 服务的 py-spy / cProfile / tracemalloc / asyncio / GIL 问题时，读取 `references/python-profiler.md`
- 需要分析吞吐、时延、丢包、IRQ、softirq、队列积压、控制面/数据面隔离、定时任务干扰时，读取 `references/network-gateway.md`
- 只有在用户明确要求保存完整结论、需要汇报或留档时，才读取 `references/report-template.md`

## 首轮响应模板

skill 触发后，优先按以下顺序行动：

1. 确认代码范围和组件边界。
2. 确认当前是“只有代码可看”，还是“已有产品环境数据可分析”。
3. 确认环境约束：是否能拿到 PID、是否能执行脚本、是否允许高扰动工具。
4. 如果没有运行数据，先做静态分析并给出带置信度的首轮判断。
5. 如果需要运行时证据，优先给低扰动采集脚本，再决定是否升级工具。

首轮回复应尽量包含：
- 已理解的问题范围
- 当前最需要的补充信息
- 先做静态分析还是先做目标环境取证
- 预期下一步产出

## 工作流程

```
阶段1: 确认代码范围与环境约束
阶段2: 做静态分析并输出首轮判断
阶段3: 按问题类型选择参考文档与采集方式
阶段4: 指导目标设备取证并回收结果
阶段5: 基于证据更新结论并继续迭代
```

## 触发示例

**应该触发**：
- “帮我看下这个 gateway 服务为什么 PPS 上不去，代码在当前仓库里，但程序只能在设备上跑。”
- “这个 Go 服务线上 CPU 很高，我只能拿到 pprof 和 /proc 信息，先帮我分析代码可能的热点。”
- “IDS 的定时任务一跑就抖，P99 延迟会尖刺，不能在开发机复现，怎么查？”
- “这个 Python 网关进程内存一直涨，我可以在目标机器上执行脚本并贴回结果。”

**不应该触发**：
- “帮我实现一个新的转发规则接口。”
- “给这个模块补单元测试。”
- “把 README 改清楚一点。”
- “只是想重构下目录结构，和性能没关系。”

## 阶段1: 代码收集

### 1.1 确认代码范围

网络设备通常有多个组件，需要先确认分析范围：
- 当前目录是否包含需要分析的代码
- 控制面、转发面、管理面是否在同一仓库
- 如不在同一仓库，是否需要提供其他代码路径

### 1.2 收集环境信息（可选）

这些信息有助于提升静态分析准确率，但不提供也可以继续。

如果用户明确说“只能在产品环境采集”或“不能在开发机运行测试程序”，优先补充以下约束信息：
- 是否能拿到目标进程 PID
- 是否有 root / sudo 权限
- 是否能执行 shell 脚本
- 是否有维护窗口或压测窗口
- 是否允许使用 perf / strace / top / taskset / ethtool
- 采集结果是否只能复制粘贴，还是可以保存成文件后回传

**运行环境**：
- CPU：型号、核数、频率、NUMA 拓扑
- 内存：大小、通道数
- 网卡：型号、队列数、驱动版本
- OS：内核版本、发行版
- 部署方式：裸机 / VM / 容器

**编译环境**：
- C: gcc/clang 版本、编译选项（-O2/-O3、-march）
- Go: go 版本、CGO_ENABLED、构建标签
- Python: python 版本、是否使用 C 扩展

**当前问题现象**：
- 具体症状：CPU 高？丢包？时延抖动？内存增长？
- 发生时机：启动后？特定操作后？流量高峰？
- 复现条件：如何触发？
- 已尝试的分析：做过哪些排查？

**信息格式**（供参考，非必须填写）：
```
运行环境：
  CPU: Intel Xeon 8380 x 2, 64C/128T, NUMA 2节点
  内存: 256GB DDR4-3200
  网卡: Intel E810-C, 25GbE, 队列数 64
  OS: CentOS 7.9, kernel 5.10

编译环境：
  GCC: 10.3.0, -O2 -march=native

问题现象：
  症状: 转发面 CPU 使用率 100%，PPS 上不去
  时机: 64 字节小包压测时
  复现: 使用测试仪发送 10Gbps 小包流量
```

## 阶段2: 代码静态分析

根据识别的语言，读取 `references/code-patterns.md` 中对应的规则，扫描代码。

### 2.1 项目识别

自动检测当前目录的项目类型：

| 检查项 | 方法 |
|--------|------|
| C 工程 | 存在 Makefile / CMakeLists.txt / *.c 文件 |
| Go 工程 | 存在 go.mod |
| Python 工程 | 存在 requirements.txt / pyproject.toml / *.py |
| 内核模块 | 存在 Kbuild / *.ko / 内核头文件引用 |

识别服务类型（需要用户确认或从代码推断）：
- **转发面**：包处理、会话表、规则匹配
- **控制面**：配置管理、路由协议、策略下发
- **管理面**：REST API、CLI、日志、监控
- **定时任务**：cron、ticker、定时扫描、规则同步

### 2.2 扫描重点

1. **热路径中的开销**：内存分配、syscall、锁、日志
2. **并发问题**：锁竞争、goroutine 泄漏、GIL 竞争
3. **内存问题**：泄漏、碎片、无界缓存、逃逸
4. **网络设备专项**：per-packet 开销、控制面干扰数据面、中断处理
5. **IO/数据库**：频繁 IO 操作、连接池配置、同步阻塞

### 2.3 输出第一版报告

静态分析完成后，输出第一版报告，仅包含代码层面的发现。

**报告内容**：
```
## 代码分析结果（第一版）

### [严重程度] 问题标题
- **位置**: 文件:行号
- **类别**: CPU / 内存 / IO / 锁 / 调度
- **现象**: 简要描述
- **代码**: 问题代码片段
- **建议**: 优化方案
- **置信度**: 高/中/低（基于代码分析，未结合运行数据）
```

**说明**：
- 第一版报告基于纯代码分析，可能包含误报
- 如用户提供了环境信息，可结合环境特点调整分析
- 标记置信度，便于后续验证

## 阶段3: 增强分析（可选）

如果用户后续提供了更准确的运行环境信息，可以：

1. **重新评估**：结合环境特点调整分析结论
2. **补充发现**：针对特定硬件/配置发现新问题
3. **更新报告**：输出增强版报告

**增强分析触发条件**：
- 用户提供了运行环境信息（CPU、内存、网卡等）
- 用户提供了编译环境信息
- 用户描述了具体问题现象

**增强分析输出**：
```
## 增强分析报告

### 环境相关发现
（结合用户提供的环境信息）

### 问题优先级调整
（根据环境特点调整建议优先级）

### 新增建议
（针对特定环境的优化建议）
```

## 阶段4: 目标设备取证指导

根据问题类型，从 `references/` 选择对应指导文档：

| 问题类型 | 参考文档 |
|----------|----------|
| CPU 热点 | c-profiler.md / go-profiler.md / python-profiler.md |
| 内存问题 | c-profiler.md / go-profiler.md / python-profiler.md |
| 吞吐/时延 | network-gateway.md |
| 定时任务 | network-gateway.md → 调度分析 |
| 锁/并发 | 对应语言 profiler 文档 |
| 网络栈 | network-gateway.md |

### 4.1 采集手段分级

默认使用低扰动、只读的采集方式，按下面顺序逐级升级：

**L1：默认手段（优先）**
- 读取 `/proc`、`/sys`、`/proc/net/*`、`/proc/irq/*`
- 使用 `scripts/` 中的采集脚本
- 适合产品环境首轮定位

**L2：中等扰动（用户确认后）**
- `top`、`ss`、`pmap`、`ethtool -S`、`taskset -p`
- 适合进一步确认热点线程、socket、内存映射和队列分布

**L3：高扰动/专项工具（仅维护窗口或用户明确允许时）**
- `perf`、`strace`、`valgrind`、长时间 profile、火焰图采样
- 必须明确提示风险：可能引入明显性能扰动、额外 CPU 开销、延迟抖动，甚至影响线上指标

### 4.2 使用要求

- 不要默认要求用户在开发机运行目标程序、压测程序或复现环境。
- 不要把 `perf`、`strace`、`valgrind` 当作第一步。
- 当只有代码、没有运行数据时，只能输出带置信度的候选问题，不要过度下结论。
- 当用户只能复制粘贴文本结果时，优先选择当前脚本和纯文本命令。

提供采集脚本（`scripts/` 目录）时：
- 先解释为什么选这个脚本
- 说明需要的输入（如 PID、采集时长）
- 说明脚本是读信息还是会引入额外负载
- 说明下一步希望用户回传哪些关键字段

**重要**：用户在目标设备上运行脚本后，将结果粘贴回来进行分析。

## 阶段5: 迭代分析

用户提供采集数据后，进入迭代分析循环：

```
用户提供采集数据
  → 分析数据
  → 更新结论
  → 提出新的采集需求（如需要）
  → 用户再次采集
  → ...
```

### 输出要求

**默认输出：终端摘要**

优先输出简洁结论，适合用户快速继续下一轮排查：

```markdown
## 本轮结论
- 主要发现：
- 当前最可能的瓶颈：
- 证据：
- 置信度：高/中/低
- 建议下一步：
```

每条结论都要尽量绑定证据来源，例如：
- 来自静态代码扫描
- 来自 `/proc/<pid>/status`
- 来自 `/proc/net/softnet_stat`
- 来自 `ethtool -S`
- 来自 pprof / perf / trace 输出

**按需输出：Markdown 报告**

只有在以下场景才生成 `performance-report.md`：
- 用户明确要求保存完整报告
- 需要跨人协作、汇报或留档
- 当前分析已经形成较完整的证据链

生成完整报告时，结构参考 `references/report-template.md`。
